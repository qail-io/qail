Because that's what actually happens in production with most ORMs. You fetch users, then loop and lazy-load posts for each one — that's N+1 by default. Libraries like ActiveRecord, Eloquent, Django ORM all do this unless you explicitly remember to eager-load. The point of the comparison isn't "look, a JOIN is faster than N+1" (obviously it is). The point is that with Qail, N+1 is structurally impossible — there's no lazy loading, no implicit per-row queries. If you want related data, you use join_on() and it's one query. You can't accidentally write the slow version because the API doesn't offer it.

We also have apples-to-apples benchmarks — same prepared statement, same query, Qail vs SQLx vs SeaORM vs libpq vs pgx. The N+1 comparison is just showing the real-world problem the architecture prevents.
